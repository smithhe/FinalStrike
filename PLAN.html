FinalStrike

A standalone Python orchestrator that mirrors only the testing workflow of Cursor cloud agents — environment bootstrap, build/lint gates, terminal tests, API checks, and computer-use UI verification — running on self-hosted GUI VMs and producing Cursor-parity evidence bundles.

North Star & Scope

Explicitly out of scope

FinalStrike mirrors testing only. It does not implement code generation, subagent spawning, CI auto-fix loops, static Bugbot/Security review, PR creation, or any other Cursor cloud agent capability outside verification.

1.1 What Cursor cloud agents do when testing (our reference model)

Based on observed Cursor cloud agent behavior, the verification loop is layered — not a single E2E framework:

  1. Orient — read repo instructions (AGENTS.md), ports, smoke shortcuts, secrets requirements
  2. Bootstrap environment — install deps, start API/DB/UI via .cursor/environment.json terminals
  3. Build & lint gate — compile, typecheck, linter before functional tests
  4. Terminal test suite — run project-native commands (dotnet test, pytest, npm test)
  5. API smoke checks — HTTP assertions against running services
  6. UI verification — computer-use: screenshot + accessibility context → plan actions → interact → screenshot evidence
  7. Gap report — explicitly list what was not tested (auth flows, live third-party APIs, etc.)
  8. Evidence bundle — logs, screenshots, desktop video, structured summary (what Cursor attaches to PRs)

Core philosophy

FinalStrike is an orchestrator + evidence recorder, not a test framework. It runs the same commands a Cursor cloud agent would run, uses an LLM to translate acceptance criteria into a verification plan, drives the UI like computer-use does, and packages proof. Repos keep their existing test stacks; FinalStrike coordinates them.

2.2 Component responsibilities

🎛️ Orchestrator (finalstrike run)

Top-level state machine. Sequences layers in order, aborts or continues based on policy, aggregates results into a single RunResult.

Python asyncio State machine

📋 Config Loader

Reads finalstrike.yaml, AGENTS.md, .cursor/environment.json. Merges secrets from vault. Validates schema.

Pydantic models

🧠 LLM Test Planner

Input: acceptance criteria markdown + repo context snapshot. Output: structured VerificationPlan with terminal commands, API checks, and UI scenarios. Uses a single OpenAI-compatible client — cloud APIs, Ollama, and gateways share one adapter.

openai_compat Structured JSON output

🖥️ Env Orchestrator

Parses environment.json install, terminals, start. Manages long-running processes, health-check URLs, tmux-like session persistence on VM.

Process supervisor

⚙️ Command Runners

Build/lint, terminal tests, API HTTP checks. Captures stdout/stderr, exit codes, duration. Supports retry policy.

Subprocess

🖱️ Computer-Use Executor

Loop: capture desktop screenshot + a11y tree → LLM action → OS input → verify. Supports browser windows and native apps.

Vision + input

🎬 Evidence Recorder

Full-desktop video (FFmpeg), per-step screenshots, command log capture. Timestamps synced to actions.

FFmpeg PNG/WebM

📊 Reporters

HTML report (primary human artifact), Slack bot uploads (screenshots/video), JSON export for future CI gates.

Configuration Contract

4.1 finalstrike.yaml (per target repo)

# finalstrike.yaml — orchestration config committed to target repo
version: "1"

project:
  name: my-app

llm:
  provider: openai_compat   # single adapter; base_url routes to any gateway
  base_url: https://api.openai.com/v1  # or http://localhost:11434/v1 (Ollama)
  model: gpt-4o              # or local model name, e.g. llama3
  # api_key via secrets vault (OPENAI_API_KEY or custom), not in file

build:
  commands:
    - name: compile
      run: dotnet build Atlas.sln
    - name: lint
      run: npm run lint
      optional: true

tests:
  commands:
    - name: unit
      run: dotnet test Atlas.sln --filter Category=Unit
    - name: integration
      run: dotnet test Atlas.sln --filter Category=Integration

api:
  base_url: http://localhost:5080
  # Specific checks derived by LLM from AC; defaults for health:
  health:
    - method: GET
      path: /health
      expect_status: 200

ui:
  base_url: http://localhost:5173
  browser: chromium       # launched on VM desktop
  smoke_route: "/#/tasks"   # fallback; AGENTS.md may override

secrets:
  file: .finalstrike/secrets.env  # gitignored; injected into env

evidence:
  video: true
  screenshots: true
  output_dir: .finalstrike/runs

slack:
  bot_token_secret: SLACK_BOT_TOKEN
  channel_id: C0123456789

policy:
  fail_fast: false            # continue layers to collect full evidence
  max_ui_steps: 40
  max_ui_retries: 4           # mirror Cursor browser MCP limit

LLM configuration examples

# OpenAI API
llm:
  provider: openai_compat
  base_url: https://api.openai.com/v1
  model: gpt-4o

# Ollama (local)
llm:
  provider: openai_compat
  base_url: http://localhost:11434/v1
  model: llama3

# LiteLLM / OpenRouter / other gateway
llm:
  provider: openai_compat
  base_url: http://localhost:4000/v1
  model: anthropic/claude-sonnet-4-20250514

4.2 AGENTS.md (optional, per target repo)

Parsed as unstructured context for the LLM planner. Recommended sections:

  • Services — what runs on which port
  • Smoke routes — URL paths that bypass auth or reach key UI fast
  • Test commands — canonical commands if not in finalstrike.yaml
  • Known gaps — integrations that cannot be automated (live Azure DevOps, etc.)

4.3 .cursor/environment.json (reused as-is)

FinalStrike reads Cursor's existing format for install, start, and terminals to avoid duplicating service definitions. Parser contract is derived from Cursor documentation and public examples with loose parsing and a version-field check. If missing, fall back to finalstrike.yaml services block (future extension). The fixture repo at fixtures/sample-app/ includes a reference environment.json for tests.

4.4 Git branch semantics

--branch records the branch name in RunResult for report context only. FinalStrike does not run git checkout on the target repo in v1 — the user ensures the working tree is on the intended branch before invoking finalstrike run.

4.5 Acceptance criteria input (runtime)

# Passed at run time — NOT committed by default
finalstrike run \
  --repo /path/to/target \
  --acceptance acceptance.md \
  --branch feature/my-change   # metadata only; checkout manually first

# acceptance.md example
## Feature: Task creation

### Acceptance Criteria
- User can open the Tasks page and see the task list
- User can click "New Task", fill title and description, and save
- New task appears in the list with correct title
- API returns 201 on POST /api/tasks with valid payload

Computer-Use Execution Layer

Approach decision deferred to Phase 5 spike

Three viable approaches — implement a spike on Linux VM before committing:

Approach Pros Cons Recommendation
A. Custom Python loop
mss screenshot + AT-SPI/a11y + pyautogui/xdotool		 Full control; no vendor lock-in; works offline with local models Most engineering effort; cross-OS input differences Primary path for Linux v1
B. Vendor computer-use API
Anthropic computer use, etc.		 Faster to prototype; model trained for desktop interaction Cost; latency; API availability; less control over evidence format Optional via gateway — if vendor exposes an OpenAI-compatible or custom endpoint, route through openai_compat base_url; spike compares in P6
C. Hybrid browser + OS
Playwright for browser a11y; pyautogui for native		 Reliable a11y tree in browser; separates concerns Two code paths; doesn't unify native app interaction Browser sub-path inside A — use Playwright CDP for a11y snapshot where a browser window is active

6.1 Computer-use action loop (mirrors Cursor)

1. Capture full-desktop screenshot → 2. Capture a11y tree → 3. LLM: next action
↓
4. Execute (click/type/key/scroll) → 5. Screenshot evidence → 6. Verify or retry (max 4) → 7. Next step or done

6.2 Action vocabulary

  • click(x, y) / click(element_ref) — coordinate or a11y-ref resolved to coordinates
  • type(text) — keyboard input to focused element
  • key(combo) — e.g. Return, Ctrl+S, Alt+Tab (for native app switching)
  • scroll(direction, amount)
  • wait(condition) — time, text visible, element present (via a11y poll)
  • launch(app) — open browser URL or native app binary
  • focus_window(title|pid) — for native desktop app context switching
  • assert_visible(description) — LLM judges screenshot against expectation

6.3 Linux implementation stack (Phase 5–6)

Session type (X11 vs Wayland) is detected at runtime; the matching driver set is selected automatically.

Screenshot

mss or grim (Wayland) / scrot (X11) for full desktop capture at planner cadence.

Accessibility

pyatspi (AT-SPI2) for Linux native + browser accessibility tree when exposed.

Input

xdotool (X11) or ydotool (Wayland) for mouse/keyboard. Abstract behind InputDriver interface.

Video

ffmpeg -f x11grab or wf-recorder (Wayland) for full-desktop WebM recording synced to run.

VM Deployment Model

FinalStrike assumes SSH (or agent daemon) access to an existing self-hosted GUI VM. No Docker — desktop apps and computer-use require a real display session.

7.1 Deployment modes

ModeDescriptionPhase
Local agent on VM FinalStrike installed directly on the GUI VM. CLI runs in user desktop session. Simplest for v1. 2+
Remote trigger CLI on dev machine SSHes to VM, starts finalstrike-worker daemon, streams results back. 9
VM pool Multiple pre-warmed VMs; orchestrator assigns idle VM per run. Requires queue. 10+

7.2 VM prerequisites checklist

  • GUI session active (X11 or Wayland) with logged-in user — FinalStrike auto-detects session type and selects drivers
  • Python 3.11+ and FFmpeg installed
  • Target repo cloned at known path
  • Secrets file deployed to .finalstrike/secrets.env (out of band)
  • Display resolution fixed (1920×1080 recommended for consistent screenshots)
  • Network access to service ports (localhost services started by env orchestrator)

7.3 Multi-OS abstraction

  • ScreenshotDriver — capture full desktop
  • AccessibilityDriver — enumerate UI tree
  • InputDriver — mouse, keyboard, window focus

Implementation Phases

Phases are ordered by dependency. Each phase ends with demonstrable, testable deliverables before the next begins.

Phase 0 — Project Foundation

P0

Dependencies: none

  • Python package scaffold (pyproject.toml, finalstrike module, Typer CLI stub)
  • Pydantic models for config, RunResult, VerificationPlan
  • JSON schemas exported to schemas/
  • Basic CLI: finalstrike --version, finalstrike validate-config
  • Sample finalstrike.yaml in fixtures/sample-app/
  • README with architecture summary linking to this plan

Exit criteria: finalstrike validate-config --repo fixtures/sample-app parses the sample finalstrike.yaml without error.

Phase 1 — Config & Context Loading

P1

Dependencies: P0

  • finalstrike.yaml loader with schema validation
  • AGENTS.md reader (raw text → planner context block)
  • .cursor/environment.json parser
  • Secrets vault loader (.finalstrike/secrets.env, env var injection)
  • Acceptance criteria markdown loader (CLI --acceptance file or --acceptance-stdin)

Exit criteria: finalstrike plan --repo X --acceptance ac.md --dry-run prints merged config + AC text.

Phase 2 — Environment Orchestrator

P2

Dependencies: P1

  • Run install command from environment.json
  • Start terminals entries as background processes with log capture
  • Health-check polling (HTTP GET until 200 or timeout)
  • Teardown: kill child processes on run end
  • Record env layer in RunResult with stdout/stderr logs

Exit criteria: On fixtures/sample-app/, finalstrike env up --repo fixtures/sample-app starts services and reports healthy.

Phase 3 — Build, Lint & Terminal Test Runners

P3

Dependencies: P2

  • Generic command runner: subprocess, timeout, stdout/stderr capture, exit code
  • Build/lint layer from finalstrike.yaml build.commands
  • Terminal test layer from tests.commands
  • Parse test output for pass/fail counts (pytest parser first; xUnit, jest parsers extensible)
  • Retry policy for flaky tests (configurable)

Exit criteria: finalstrike run --repo X --acceptance ac.md --layers build,terminal produces RunResult JSON with test counts.

Phase 4 — API Check Runner

P4

Dependencies: P2

  • HTTP client for GET/POST/PUT/DELETE with headers, body, auth from secrets
  • Health checks from finalstrike.yaml api.health
  • LLM-planned API checks from VerificationPlan
  • Response assertions: status code, JSON path values, header presence
  • Record request/response in RunResult (redact secrets)

Exit criteria: API layer runs against a live local API and reports pass/fail per check.

Phase 5 — LLM Test Planner

P5

Dependencies: P1

  • openai_compat provider adapter (OpenAI Python SDK + configurable base_url / api_key)
  • Prompt template: AC markdown + AGENTS.md + finalstrike.yaml → VerificationPlan JSON
  • Schema validation of LLM output; retry on malformed JSON
  • finalstrike plan command outputs plan without executing
  • Gap analysis section in plan output
  • Document example configs for OpenAI API, Ollama (localhost:11434/v1), and gateway proxies

Exit criteria: Given sample AC markdown, planner produces valid VerificationPlan mapping criteria to terminal, API, and UI layers.

Phase 6 — Computer-Use Spike (Linux)

P6

Dependencies: P5

  • Implement ScreenshotDriver, AccessibilityDriver, InputDriver for Linux
  • Auto-detect X11 vs Wayland; select xdotool/ydotool and screenshot/video backend accordingly
  • Single-scenario proof: "open browser to URL, verify page title visible"
  • Compare Approach A vs B (custom vs vendor API) — document decision
  • Per-step screenshot saved with timestamp and action metadata
  • Enforce max 4 retries per action (Cursor parity)

Exit criteria: One UI scenario from VerificationPlan executes end-to-end on Linux VM with screenshot evidence.

Phase 7 — Evidence Recorder & Gap Analyzer

P7

Dependencies: P3, P4, P6

  • Full-desktop FFmpeg video recording for entire run
  • Artifact directory structure: .finalstrike/runs/<run_id>/
  • Sync video timestamps to UI step markers
  • Gap analyzer: merge planner gaps + runtime failures + policy exclusions
  • RunResult JSON as canonical machine output

Exit criteria: Full run produces video + screenshots + RunResult JSON matching schema.

Phase 8 — HTML Report Generator

P8

Dependencies: P7

  • Jinja2 template: summary header, per-layer sections, embedded screenshots
  • Video player embed (or link) for desktop recording
  • Terminal output collapsible blocks with syntax highlighting
  • Gap report section styled prominently
  • Cursor-parity layout: status badge, test counts, evidence gallery, logs

Exit criteria: HTML report is self-contained and viewable in browser; matches Cursor PR artifact information density.

Phase 9 — Slack Bot Reporter

P9

Dependencies: P8 — deferred until HTML reports are working; not required for first usable milestone

  • Slack bot token auth from secrets vault
  • Summary message with pass/fail/partial status
  • Upload screenshots and video via files.upload (or v2 API equivalent)
  • Link to HTML report if hosted; else attach key screenshots inline
  • Configurable channel per finalstrike.yaml

Exit criteria: Run completion posts to Slack with video + screenshot attachments.

Phase 10 — Full Orchestrator Integration

P10

Dependencies: P2–P9

  • finalstrike run executes full layer sequence: env → build → terminal → api → ui
  • Policy: fail_fast vs collect-all-evidence modes
  • CLI flags: --layers, --skip-env, --plan-only, --no-slack
  • --branch records metadata only; no automatic git checkout
  • Exit code 0 only when all executed layers pass
  • End-to-end demo on fixtures/sample-app/

Exit criteria: Single command produces Cursor-parity evidence bundle from AC markdown input.

Phase 11 — Windows Platform Support

P11

Dependencies: P10

  • Windows ScreenshotDriver, AccessibilityDriver (UIA), InputDriver (pywinauto)

  • FFmpeg gdigrab video recording

Phase 12 — macOS Platform Support

P12

CLI Interface

# Validate repo config
finalstrike validate-config --repo /path/to/target

# Generate verification plan from acceptance criteria (no execution)
finalstrike plan --repo /path/to/target --acceptance acceptance.md

# Run specific layers
finalstrike run --repo /path/to/target --acceptance acceptance.md --layers env,build,terminal

# Full verification run
finalstrike run --repo /path/to/target --acceptance acceptance.md --branch feature/x  # branch = metadata only

# Environment management only
finalstrike env up --repo /path/to/target
finalstrike env down --repo /path/to/target

# Accept AC from stdin (e.g. pasted PR body)
gh pr view 123 --json body -q .body | finalstrike run --repo . --acceptance-stdin

Slack Evidence Delivery

Deferred until P8

Slack reporting is implemented in Phase 9, after the HTML report generator is complete. The first usable milestone is a self-contained HTML evidence bundle; Slack is optional via --no-slack once implemented.

10.1 Message structure

  1. Header — ✅/❌/⚠️ status, repo name, branch, run ID, duration
  2. Summary blocks — per-layer pass/fail with counts (e.g. "138 tests passed")
  3. Gap warning — bullet list of untested items
  4. Attachments — desktop video (WebM), key screenshots (first, last, failures)
  5. Footer — link to full HTML report path on VM or hosted URL

10.2 Setup requirements

  • Slack app with files:write, chat:write scopes
  • Bot token stored in secrets vault
  • Channel ID in finalstrike.yaml
